### 回答 (2026-05-03) — 辻ラインの時間解像度を10秒間隔に向上

#### 調査結果

辻ライン (DPライン) のコードを精査したところ、`toFixed` などの精度を落とす処理は見当たりませんでした。原因は **時間サンプリングの解像度** でした。

- **辻検索**: 1分間隔スキャン + ベスト1分の前後60秒を**1秒間隔でリファイン**(`tsuji-search-worker.js`)
- **辻ライン (旧)**: **1分間隔のみ** でサンプリング (`calculateDPPathPoints`)

例えば 辻検索 の最良マッチが 12:34:23 だったとき、辻ラインは 12:34:00 と 12:35:00 の2点を線分で結ぶだけなので、その間の23秒分(約100m〜数百m)の位置誤差が生じていました。太陽は約 0.25°/分 動くため、1分間で数百メートルもの位置誤差になります。

#### 修正内容

`calculateDPPathPoints()` のサンプリング間隔を **1分 → 10秒** に変更しました。

```js
for (let m = 0; m < 8640; m += 1) { // 10秒毎 (1日 = 8640 × 10秒)
    const time = new Date(startOfDay.getTime() + m * 10000);
```

これにより、線の各点がより密になり、観測点位置と辻検索結果の間のズレが大幅に縮小します。

また、5分マーカーの条件を `getMinutes() % 5 === 0` だけだと10秒ごとに引っかかってしまうため、`getSeconds() === 0` を追加しました。

#### パフォーマンスへの影響

- 旧: 1日 1440 回 × 3日 × 表示天体数 (例: 5体) = 21,600 回の計算
- 新: 1日 8640 回 × 3日 × 5体 = 129,600 回 (6倍)

`Astronomy.Horizon` の計算はそれほど重くないため、実用的に問題ない範囲だと考えますが、もし動作が遅くなるようでしたらお知らせください (例: 30秒間隔に下げる調整等を検討します)。
