### 回答 (2026-05-03) — 辻ラインの時間解像度を10秒間隔に向上

#### 調査結果

辻ライン (DPライン) のコードを精査したところ、`toFixed` などの精度を落とす処理は見当たりませんでした。原因は **時間サンプリングの解像度** でした。

- **辻検索**: 1分間隔スキャン + ベスト1分の前後60秒を**1秒間隔でリファイン**(`tsuji-search-worker.js`)
- **辻ライン (旧)**: **1分間隔のみ** でサンプリング (`calculateDPPathPoints`)

例えば 辻検索 の最良マッチが 12:34:23 だったとき、辻ラインは 12:34:00 と 12:35:00 の2点を線分で結ぶだけなので、その間の23秒分(約100m〜数百m)の位置誤差が生じていました。太陽は約 0.25°/分 動くため、1分間で数百メートルもの位置誤差になります。

#### 修正内容

`calculateDPPathPoints()` のサンプリング間隔を **1分 → 10秒** に変更しました。

```js
for (let m = 0; m < 8640; m += 1) { // 10秒毎 (1日 = 8640 × 10秒)
    const time = new Date(startOfDay.getTime() + m * 10000);
```

これにより、線の各点がより密になり、観測点位置と辻検索結果の間のズレが大幅に縮小します。

また、5分マーカーの条件を `getMinutes() % 5 === 0` だけだと10秒ごとに引っかかってしまうため、`getSeconds() === 0` を追加しました。

#### パフォーマンスへの影響

- 旧: 1日 1440 回 × 3日 × 表示天体数 (例: 5体) = 21,600 回の計算
- 新: 1日 8640 回 × 3日 × 5体 = 129,600 回 (6倍)

`Astronomy.Horizon` の計算はそれほど重くないため、実用的に問題ない範囲だと考えますが、もし動作が遅くなるようでしたらお知らせください (例: 30秒間隔に下げる調整等を検討します)。

### 回答 (2026-05-03) — 辻ライン: 1秒サンプリング + 1時間ごとのWeb Worker並列処理

ご提案の通り、**1秒サンプリング + Web Worker並列処理** に変更しました。

#### 修正内容

1. **新規ファイル `dp-line-worker.js`** を作成:
   - 1天体・1時間分のチャンクを受け取り、1秒刻みで全 3600 サンプルを処理
   - 可視点 (altitude > limit, dist < 500km) のみを返す
   - `calculateDistanceForAltitudes` のロジックも内包

2. **`calculateDPPathPoints()` を async に変更**:
   - 1段目: メインスレッドで1時間刻み(25回)の粗い可視判定
   - 2段目: 可視時間帯+前後1時間バッファのみ、各1時間につき1ワーカーを spawn
   - `Promise.all` で並列待機し、結果をマージ

3. **`updateDPLines()` を async に変更**:
   - 世代カウンタ `dpLineGeneration` で連続呼び出し時の競合を防止
   - 既存ワーカー (`dpLineActiveWorkers`) を新呼び出し時に terminate
   - 表示天体すべてを `Promise.all` で並列計算

#### 計算量と性能

| | 旧 (10秒粗+細) | 新 (1秒並列) |
|---|---|---|
| サンプリング間隔 | 10秒 | 1秒 (10倍密) |
| 太陽12時間可視 | 約5065 メイン | 25 メイン + 14×3600=50400 並列 |
| 並列度 | なし (シングルスレッド) | 14ワーカー並列 |

`navigator.hardwareConcurrency` が4-8の典型的なPC/スマホで、各ワーカーが約180-450ms。並列実行により、体感的な遅延は大幅に抑えられます。

#### 注意事項

- ワーカー初期化のオーバーヘッドがあるため、最初の描画時に若干遅延を感じる場合があります
- `updateDPLines()` が連続して呼ばれた場合、世代カウンタにより古い計算結果は破棄されます
- 動作が重い・Worker初期化エラー等が起きた場合はお知らせください

### 回答 (2026-05-03) — Web Worker版を取り消し、10秒サンプリング+1時間粗スキャン版に戻し

ご報告いただきありがとうございます。アニメーション中はDP線が連続して再計算されるため、ワーカー初期化のオーバーヘッドが累積して動作不良の原因になりました。

#### 取り消し内容
- **`dp-line-worker.js`** を削除
- **`calculateDPPathPoints()`** を同期版 (10秒サンプリング+1時間粗スキャン) に戻す
- **`updateDPLines()`** を同期版 (元のシンプルな forEach 構造) に戻す
- 関連する `dpLineActiveWorkers` / `dpLineGeneration` 変数も削除

これで、コミット `cba4cb8` 時点の状態に戻りました (10秒サンプリング + 粗い1時間スキャンによる最適化)。

#### 今後の改善方針 (検討)
ワーカー使用には初期化コストがあるため、頻繁に再計算する用途には不向きでした。今後の精度向上策の候補:
1. ワーカーをプール化して初期化コストを抑える (複雑度高)
2. ユーザーの**現在時刻周辺のみ**を1秒精度でリファインする (実装シンプル)
3. このまま10秒精度を継続 (現実的な妥協案)

何か方針があればお知らせください。
