## リポジトリとブランチの違い

### リポジトリ = 独立したプロジェクトの箱

リポジトリは「プロジェクト全体を入れる箱」。
異なるリポジトリ間はお互いに独立しており、直接の連携はない（手動またはCI/CDで同期する）。

```
リポジトリA (soranotsuji)       リポジトリB (soranotsuji-dev)
┌─────────────────────┐       ┌─────────────────────┐
│  独自の履歴          │       │  独自の履歴          │
│  独自の設定          │       │  独自の設定          │
│  独自のブランチ群    │       │  独自のブランチ群    │
└─────────────────────┘       └─────────────────────┘
       ↑ 独立               ↑ 独立
       （直接の連携なし。手動またはCI/CDで同期）
```

**リポジトリを分ける理由:**
- 本番と開発で**アクセス権限**を分けたい（本番は慎重に管理）
- GitHub Pagesの**ドメイン**が異なる（soranotsuji.net vs github.io/soranotsuji-dev）
- 本番に不要なブランチやコミット履歴を持ち込まない

### ブランチ = 同じリポジトリ内の作業の分岐

ブランチは「同じ箱の中での作業の枝分かれ」。
同じリポジトリ内なので、簡単に統合（merge）できる。

```
同じリポジトリ内:
main:        A → B → C ──────────────────→ M（マージコミット）
                  ↘                        ↗
develop:       D → E → F → G ──────────┘
                       ↘
work:              W1 → W2（たけちゃんの修正）
                       ↘
claude/yyy:        H → I（Claudeの作業中）
```

**ブランチを分ける理由:**
- **main**: 安定版。いつでもデプロイ可能な状態を保つ
- **develop**: テスト済みの統合コード。GitHub Pagesで公開
- **work**: たけちゃんの確認・調整用
- **claude/yyy**: 開発者Claudeの個別作業

## リポジトリ・ブランチ構成

### リポジトリ構成

```
┌──────────────────────────────────────────────────┐
│ soranotsuji（本番環境）                            │
│   ブランチ: main のみ                              │
│   用途: soranotsuji.net でユーザーに公開           │
│   タグ: v1.17.0, v1.18.0, ...                     │
└──────────────────────────────────────────────────┘
       ↑ デプロイ（手動 or 自動）
       │
┌──────────────────────────────────────────────────┐
│ soranotsuji-dev（開発環境）                        │
│   ブランチ: main, develop, work, claude/yyy       │
│   用途: 開発・テスト                               │
│   タグ: v1.17.0, v1.18.0, ...                     │
│   GitHub Pages: develop ブランチで動作確認         │
└──────────────────────────────────────────────────┘
       ↑ 必要に応じてpull
       │
┌──────────────────────────────────────────────────┐
│ soranotsuji-sandbox（試行環境）                    │
│   ブランチ: main, develop, work, claude/xxx       │
│   用途: 実験的な機能開発・検証                      │
└──────────────────────────────────────────────────┘
```

### 開発環境のブランチの流れ

```
claude/yyy ──→ work ←──→ claude/yyy     develop          main
 （開発）      （確認・調整）              （テスト）        （安定版）
    |              |                        |                |
    └──┬───────────┘                        |                |
       |                                    |                |
       └── 固まったら ──→ develop にマージ ──┘                |
                          |                                  |
                    テスターClaudeがテスト                     |
                    （Antigravity + Playwright）              |
                          |                                  |
                       テストOK                               |
                          |                                  |
                       main にマージ ────────────────────→ 本番デプロイ
```

### GitHub Pages の設定

| リポジトリ | GitHub Pages のブランチ | URL |
|---|---|---|
| soranotsuji | main | https://soranotsuji.net |
| soranotsuji-dev | develop | https://takeyosui15.github.io/soranotsuji-dev/ |

**ポイント:** soranotsuji-devのGitHub Pagesは**developブランチ**に設定する。
テスターClaude（Antigravity）がこのURLでPlaywrightテストを実行する。

## 日常の作業フロー

### 各ブランチの役割と担当者

| ブランチ | 担当者 | 役割 | 確認方法 |
|---|---|---|---|
| `claude/yyy` | 開発者Claude（Claude Code） | 機能開発・バグ修正 | — |
| `work` | たけちゃん | 確認・調整・小さな修正 | ローカルサーバーでブラウザ確認 |
| `develop` | テスターClaude（Antigravity） | 統合テスト・品質確認 | GitHub Pages + Playwright |
| `main` | たけちゃん | 安定版・デプロイ準備 | — |

### 作業の流れ（詳細）

```
1. たけちゃんが依頼 → 開発者Claudeが claude/yyy で作業
2. 開発者Claudeの成果を たけちゃんが work で確認・調整
3. work ←→ claude/yyy で何往復かやりとり
4. 固まったら develop にマージ
5. テスターClaude（Antigravity）が develop でPlaywrightテスト
6. テストOK → main にマージ → 本番デプロイ
```

### 作業の種類別ルール

| 作業の種類 | 作業者 | ブランチ | 例 |
|---|---|---|---|
| 機能開発・バグ修正 | 開発者Claude | claude/yyy | 辻検索の改善、新メニュー追加 |
| 確認・小さな修正 | たけちゃん | work | ヘルプ文言修正、ボタン幅調整、色変更 |
| 統合テスト | テスターClaude | develop | Playwrightテスト実行 |
| デプロイ準備 | たけちゃん | develop → main | マージ＆タグ付け |

### 開発者Claudeとたけちゃんの作業が並行する場合

```
claude/yyy:  ... → C1 → C2 → C3
                                  ↘
work:        ... → W1 → W2 → W3 → M1（claude/yyyからマージ）
                                        ↘
                                         → M2（workからdevelopにマージ）
                                              ↘
develop:     ... → D1 ────────────────────── → M2
```

1. 開発者Claudeは claude/yyy で作業
2. たけちゃんは work で確認・修正
3. claude/yyy の成果を work にマージして確認
4. 固まったら work から develop にマージ
5. テスターClaude が develop でテスト

## Claude Code on Web でのセッション管理

### 新規セッション毎にブランチが増える

Claude Code on Webでは、新規セッション毎に新しい `claude/xxx` ブランチが**自動作成される**。
タスク定義の仕組み上、既存ブランチへのチェックアウトを指示しても403エラーで弾かれるため、
これは回避できない。

### 新規セッション開始時の指示

前回の作業内容を引き継ぐために、以下のメッセージで始める:

```markdown
order.md、todo.md を読んで、前回の作業状態を把握してください。
未解決の質問や気になる点があれば、教えてください。
```

### 不要ブランチの定期削除

作業完了後、不要になった `claude/xxx` ブランチはまとめて削除する:

```bash
# GitHub上で削除: リポジトリ → Branches → 不要なブランチを Delete

# ローカルで削除する場合:
git branch -d claude/xxx
git push origin --delete claude/xxx
```

（参照: order-to-me.md）

## タグについて

### タグはコミットに付く（ブランチではない）

タグは「特定のコミット」に対して付けるもの。ブランチ単位ではない。

```
コミット履歴:
A → B → C → D → E → F
              ↑         ↑
           v1.16.0   v1.17.0  ← タグはコミットに付く
```

### ブランチとの関係

コミットは複数のブランチに含まれることがある。
タグが付いたコミットがmainとdevelopの両方に含まれていれば、
どちらのブランチからもそのタグが見える。

```
main:      A → B → C → D ← v1.17.0
                        ↑
develop:   A → B → C → D → E → F（開発中）
                        ↑
                     v1.17.0 はmainにもdevelopにも見える
                    （コミットDに付いているから）
```

### タグを付けるタイミング

- **mainにマージした時点**でタグを付ける
- developやclaude/yyyにはタグを付けない（安定版ではないため）
- 開発環境と本番環境の両リポジトリに同じタグを付ける

```bash
# 開発環境のmainでタグを付ける
cd soranotsuji-dev-local
git checkout main
git tag v1.18.0
git push origin v1.18.0

# 本番環境にデプロイ後、同じタグを付ける
# （deploy-guide.md の手順に従う）
```

## 自動デプロイ（develop → main → 本番）の検討

### メリット
- 手動ミスの排除
- デプロイの高速化
- 一貫したプロセス

### デメリット・発生しやすいミス

| リスク | 説明 | 対策 |
|---|---|---|
| 意図しないデプロイ | mainに誤ってマージ → 即座に本番反映 | マージ前にPull Requestでレビュー |
| テスト不足 | developでのテストが不十分なままmainへ | テスターClaudeのテストを必須化 |
| 環境差異の問題 | CNAME等の差異管理がCI/CDに依存 | ワークフロー内で明示的に管理 |
| ロールバックの複雑さ | 自動化されているほど手動介入が難しい | ロールバック手順も自動化 |

### 推奨: 段階的に導入

**Phase 1（現在）:** 手動デプロイ
- deploy-guide.md の手順に従う
- 慣れるまでは手動で確実に

**Phase 2（慣れてきたら）:** 半自動デプロイ
- mainへのマージは手動（Pull Request）
- マージ後の本番デプロイはGitHub Actionsで自動化
- CNAMEの管理もワークフローに含める

**Phase 3（十分に運用が安定したら）:** 完全自動
- developからmainへのマージもPull Request + 自動マージ
- ただし個人プロジェクトではPhase 2で十分

## SSoT（シングル・ソース・オブ・トゥルース）

### 何を「正」とするか

| 状況 | 正（SSoT） | 理由 |
|---|---|---|
| 通常運用時 | soranotsuji-dev/main | 最新の安定コードがここにある |
| 本番で障害発生時 | soranotsuji/main の前のバージョン | 本番で動いていた実績がある |
| 開発中の最新コード | soranotsuji-dev/develop | 最新の開発状態 |

### 基本原則

**soranotsuji-dev/main が常にSSoT（唯一の正）。**

理由:
- 本番（soranotsuji/main）は開発環境のmainの**コピー**に過ぎない
- 開発環境のmainに全ての安定版コミットが時系列で残っている
- タグでバージョンを管理しているので、任意の時点に戻れる

```
soranotsuji-dev/main（SSoT）:  A → B → C → D → E → F
                                        ↑         ↑
                                     v1.16.0   v1.17.0

soranotsuji/main（コピー）:    A' → B' → C' → D' → E' → F'
                                        ↑          ↑
                                     v1.16.0    v1.17.0
```

### 不整合が発生した場合

1. **開発環境のブランチが削除された場合:**
   - mainが残っていれば問題なし（mainがSSoT）
   - develop、work、claude/yyyが消えても、mainから再作成可能

2. **本番環境に問題が発生した場合:**
   - 開発環境のmainから再デプロイすれば復旧できる

3. **開発環境のmainが壊れた場合（最悪のケース）:**
   - 本番環境のmainから復旧可能（本番は直近の安定版のコピー）
   - タグ付きのコミットを基に復元

## GitHub料金の確認方法

### 確認手順

1. GitHubにログイン
2. 右上のプロフィールアイコンをクリック
3. **Settings**（設定）をクリック
4. 左メニューの **Billing and plans**（請求とプラン）をクリック
5. 現在のプラン・使用状況・請求履歴が表示される

直接URL: https://github.com/settings/billing/summary

### 無料プラン（GitHub Free）の範囲

| 項目 | 無料枠 |
|---|---|
| パブリックリポジトリ | 無制限 |
| プライベートリポジトリ | 無制限（共同編集者は制限あり） |
| GitHub Pages | パブリックリポジトリで無料 |
| GitHub Actions | 月2,000分（パブリックリポジトリは無制限） |
| ストレージ | 500MB（GitHub Packages） |

**現在のプロジェクト（パブリックリポジトリ + GitHub Pages）は無料枠内で運用可能。**
GitHub Actionsを使う場合もパブリックリポジトリなら無制限。
