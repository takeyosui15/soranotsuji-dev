# 設計仕様書の書き方ガイド

機能を実装した後（または実装しながら）、「このプログラムはどう作られているか」
「どんな制約があるか」を記録するためのドキュメントです。

---

## 要件仕様との違い

| | 要件仕様 (`spec-*.md`) | 設計仕様 (`design-*.md`) |
|---|---|---|
| **いつ書く** | 作る前 | 作った後、または作りながら |
| **誰のため** | 依頼者・ユーザー | 開発者・将来の自分 |
| **書く内容** | 何がしたいか | どう作られているか |
| **例** | 「CSVでダウンロードしたい」 | 「UTF-8 BOM付き、カラムは6列」 |

---

## 設計仕様書を書くべき場面

すべての機能に必要なわけではありません。以下に該当するときに書くと効果的です。

- **データ形式を定義する機能** — CSV、JSON、LocalStorageのスキーマなど
- **外部との接点がある機能** — API連携、ファイル入出力
- **複雑な制約がある機能** — 計算精度、パフォーマンス、ブラウザ互換性
- **他の人に引き継ぐ可能性がある機能**

逆に、UIの色を変えるだけ、ボタンのラベルを変えるだけ、といった変更には不要です。

---

## 書き方のポイント

### 1. 事実を書く（意図や感想ではなく）

| 良い例 | 避けたい例 |
|---|---|
| 「文字コードはUTF-8（BOM付き）」 | 「文字コードはちゃんとしている」 |
| 「最大1000行まで対応」 | 「大量データにも対応」 |
| 「Leafletのv1.9.4 APIを使用」 | 「地図ライブラリを使用」 |

### 2. 「なぜそうしたか」を制約・判断に添える

仕様だけでなく、その背景にある理由を一言添えると、将来の変更時に役立ちます。

```markdown
- 文字コード: UTF-8（BOM付き）
  - 理由: BOMなしだとExcelで開いた際に文字化けするため
```

### 3. 関連する既存機能との関係を明記する

このプロジェクトは単一の `script.js` に機能が集約されているため、
新機能が既存のどの部分に影響するかを書いておくと重要です。

```markdown
## 既存機能との関係
- LocalStorageの `soranotsuji_app` キーに保存する設定と競合しない
- `appState.searchResults` のデータ構造をそのまま利用する
```

## ファイルの置き場所と命名

```
docs/
├── how-to-write-spec.md          # 要件仕様の書き方ガイド
├── how-to-write-design.md        # このファイル（設計仕様の書き方ガイド）
├── design-template.md            # 設計仕様のテンプレート
├── spec-csv.md                   # 要件仕様の例
└── design-csv.md                 # 設計仕様の例
```

命名規則: `design-機能名.md`

---

## テンプレートの使い方

`docs/design-template.md` をコピーして `docs/design-機能名.md` として保存し、
該当するセクションを埋めてください。不要なセクションは削除して構いません。
