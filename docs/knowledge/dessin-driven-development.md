# デッサン駆動開発（Dessin-Driven Development）

画家がキャンバスに向かう前にデッサンをするように、
コードを書く前に構想をスケッチし、対象をよく観察してから作る開発スタイル。

> 「ハッカーと画家は似ている。どちらもものを作る人間だ。」
> — ポール・グレアム

---

## このスタイルの特徴

- **デッサン（構想スケッチ）から始める** — いきなりコードを書かない
- **モデルをよく見つめる** — 作る対象を調査・観察してから描く
- **小さく回す** — 完璧を目指さず、1サイクルを軽く回して改善する
- **AIと対話しながら進める** — 一人で抱えず、AIをパートナーにする
- **ドキュメントで伝える** — 口頭ではなく、書いたもので意図を共有する

---

## 4つのドキュメント

デッサン駆動開発では、4つのドキュメントを使ってサイクルを回します。

```
docs/
├── research.md    ① 観察する
├── dessin.md      ② 描く
├── pdca.md        ③ 振り返る
└── knowledge/     ④ 蓄える
```

| ドキュメント | 役割 | 画家で言うと |
|---|---|---|
| `research.md` | 調査依頼と結果 | モデルをじっくり見つめる |
| `dessin.md` | 構想メモ（要件+設計のスケッチ） | 鉛筆でデッサンを描く |
| `pdca.md` | 改善サイクルの記録 | 描いた絵を離れて眺める |
| `knowledge/` | ナレッジの蓄積 | 技法書・参考資料を整理する |

---

## 開発の進め方

### ステップ1: 観察する（Research）

何を作るかを決める前に、対象をよく見ます。

- 知りたいことを `research.md` に書いて調査を依頼する
- 技術的な制約、既存の仕組み、ユーザーの状況を調べる
- 調査結果を同じファイルに記録する

```markdown
<!-- research.md の例 -->
## 調査: CSVの文字コード
### 知りたいこと
- Excelで文字化けしない文字コードは？
### 結果
- UTF-8（BOM付き）が必要
```

> **画家のたとえ**: モデルの輪郭、光の当たり方、影の落ち方をよく観察する段階。

### ステップ2: 描く（Dessin）

観察した結果をもとに、作りたいものの構想をスケッチします。

- `dessin.md` に簡単なメモとして書く
- 要件と設計を分けず、思いついたことをそのまま描く
- 完璧でなくていい。ラフスケッチで十分

```markdown
<!-- dessin.md の例 -->
## CSV出力
- 検索結果をCSVでダウンロードしたい
- カラム: 日時、場所、方位角、高度
- ボタンは検索結果の下に置く
- 文字コードはUTF-8 BOM付き（research参照）
```

> **画家のたとえ**: 鉛筆で大まかな構図を描く段階。細部はまだ描かない。

### ステップ3: 作る（Create）

デッサンをもとに、AIと対話しながら実装します。

- `dessin.md` をAIに共有して、意図を伝える
- AIがコードを書き、動くものを作る
- 疑問があればその場で対話して解決する

> **画家のたとえ**: 絵の具を使って本描きする段階。デッサンがあるから迷わない。

### ステップ4: 振り返る（PDCA）

作った結果を振り返り、次のサイクルにつなげます。

- `pdca.md` に Plan / Do / Check / Action を記録する
- うまくいったこと、改善したいことを書く
- Action が次のサイクルの Research や Dessin のきっかけになる

```markdown
<!-- pdca.md の例 -->
## サイクル1: CSV出力（2026-03-09）
### Plan
- 検索結果をCSVで出力する機能を追加
### Do
- dessin.mdの構想をもとに実装
### Check
- 動作OK。ただしカラム名が英語のままだった
### Action
- 次のサイクルでカラム名を日本語に変更
```

> **画家のたとえ**: 一歩引いて絵を眺める。バランスや色味を確認して、次の筆を決める。

### ステップ5: 蓄える（Knowledge）

繰り返す中で得た知識を `knowledge/` に整理します。

- 書き方ガイド、テンプレート、リファレンスなど
- 同じことを何度も調べなくて済むように
- 新しい発見があれば随時追加

> **画家のたとえ**: 技法書をまとめ、画材の特性を記録する。次の作品に活かすために。

---

## サイクルの全体像

```
    ┌─────────────────────────────────────┐
    │                                     │
    ▼                                     │
 ① 観察する ──→ ② 描く ──→ ③ 作る ──→ ④ 振り返る
  research      dessin      create       pdca
    ▲                                     │
    │                                     │
    └──────── ⑤ 蓄える ←─────────────────┘
              knowledge
```

1サイクルの大きさは自由です。
大きな機能を1サイクルで作ることもあれば、
小さな改善を何回も回すこともあります。

---

## アジャイルやウォーターフォールとの違い

| | ウォーターフォール | アジャイル | デッサン駆動開発 |
|---|---|---|---|
| 計画の粒度 | 詳細な仕様書 | ユーザーストーリー | デッサン（ラフスケッチ） |
| チーム規模 | 大規模 | 小〜中規模 | 1人 + AI |
| 反復の単位 | フェーズ | スプリント（1〜2週） | 自由（1日でも1時間でも） |
| ドキュメント | 多い・厳格 | 最小限 | 対話と記録のバランス |
| 重視すること | 計画通りに進める | 変化に対応する | よく観察してから作る |

デッサン駆動開発は、専門的なフレームワークではありません。
個人とAIが一緒にものづくりをするための、実践的な進め方です。

---

## 始め方

1. `research.md` に知りたいことを書く
2. 調査結果をもとに `dessin.md` に構想を描く
3. デッサンをAIに渡して一緒に作る
4. `pdca.md` に振り返りを書く
5. 繰り返す
