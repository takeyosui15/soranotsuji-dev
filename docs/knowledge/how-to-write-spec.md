# 仕様書の書き方ガイド

Claudeに機能の実装を依頼する際、仕様書を `.md` ファイルにまとめておくと効率的です。
このドキュメントでは、仕様書の書き方と `.md` ファイルの活用方法を説明します。

---

## 仕様書をファイルに書くメリット

- **チャットに貼ると流れてしまう** — ファイルなら何度でも参照できる
- **Markdownで構造化できる** — 見出し・箇条書き・表で整理しやすい
- **Gitで履歴管理できる** — 仕様の変遷が追える
- **修正が簡単** — チャットで再送する必要がなく、ファイルを編集するだけで済む

---

## 仕様書テンプレート

以下をコピーして `docs/spec-機能名.md` として保存し、内容を埋めてください。

```markdown
# 機能名（例: 辻カレンダー機能）

## 背景・目的
- なぜこの機能が必要か（1〜3行）

## やりたいこと
- ユーザー視点で「何ができるようになるか」を箇条書き
- 例: 「指定した月のダイヤモンド富士の日をカレンダー表示したい」

## 具体的な動作イメージ
1. ユーザーが○○ボタンを押す
2. △△が表示される
3. □□を選択すると××になる

## 画面イメージ（あれば）
- 簡単なASCII図やスクリーンショット

## 制約・注意点
- 既存の○○機能との兼ね合い
- パフォーマンス要件など
```

---

## 書き方のポイント

### 「何をしたいか」を書く（「どう実装するか」は不要）

実装方法はClaudeが提案できるので、仕様書では **ユーザー視点の要件** に集中してください。

| 良い例 | 避けたい例 |
|---|---|
| 「辻の日をカレンダーで一覧表示したい」 | 「divタグでカレンダーを作って...」 |
| 「日付をタップすると詳細が見たい」 | 「onClickイベントでモーダルを開いて...」 |

### 長くなりすぎたら分割する

1つのファイルが長くなったら、機能ごとに分けましょう。

```
docs/
├── spec-calendar.md        # カレンダー機能の仕様
├── spec-notification.md    # 通知機能の仕様
└── spec-export.md          # エクスポート機能の仕様
```

### 優先度や段階を書いておくと便利

```markdown
## 実装の優先度
1. **必須**: カレンダーの基本表示
2. **できれば**: 月齢アイコンの表示
3. **将来**: 他ユーザーとの共有機能
```

## `.md` ファイルの使い分け

| ファイル | 用途 | 読む人 |
|---|---|---|
| `CLAUDE.md` | プロジェクト構成・規約など、Claudeが毎回参照する情報 | Claude |
| `README.md` | プロジェクトの概要・使い方 | 人間（GitHub訪問者など） |
| `docs/spec-*.md` | 機能の仕様書（「こういうものを作りたい」） | Claude + 自分 |
| `docs/how-to-write-spec.md` | このファイル（仕様書の書き方ガイド） | 自分 |

---

## 実際の使い方の流れ

1. `docs/spec-新機能.md` に仕様を書く
2. Claudeとの会話で **「`docs/spec-新機能.md` を読んで実装してください」** と伝える
3. Claudeがファイルを読み、内容を理解した上で実装を提案・実行する
4. 仕様に変更があればファイルを編集し、再度Claudeに伝える
